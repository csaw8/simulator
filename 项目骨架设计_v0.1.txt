项目骨架设计 v0.1

一、文档目标

本文档用于定义项目初版的工程骨架，包括目录结构、模块边界、运行入口、核心数据流、配置方式以及后续扩展接口。

目标不是一次性定死所有实现细节，而是建立一个足够稳定、足够解耦、足够适合文本世界原型开发的工程起点。

本文档以前置文档为基础：

1. 世界模型草案_v0.1.txt
2. 运行循环草案_v0.1.txt


二、初版工程目标

| 目标 | 说明 |
|---|---|
| 先跑通文本世界闭环 | 不追求完整功能，先让世界可推进、可观察、可输出事件流 |
| 保持模块解耦 | 世界状态、运行循环、AI 调度、叙事输出分离 |
| 保留替换空间 | 未来可以更换模型、规则包、存储方式、界面形式 |
| 默认纯文本 | 初版不做复杂 UI，只做终端/文本观察台 |
| AI 可缺席 | 即便 AI 模块失败，世界仍应继续运行 |


三、技术方向建议

结合当前项目阶段，建议初版以 Python 为主实现原型。

理由如下：

| 原因 | 说明 |
|---|---|
| 迭代快 | 世界规则、角色调度、事件结构会频繁调整 |
| 文本处理方便 | 构造 prompt、事件摘要、结构化输出更直接 |
| 存储与配置轻量 | 适合先用 JSON/YAML/SQLite 等方式搭建 |
| 原型优先 | 当前目标是验证世界是否“活起来”，而不是优化极限性能 |

初版推荐组合：

| 层面 | 建议 |
|---|---|
| 语言 | Python |
| 配置 | YAML 或 JSON |
| 存储 | SQLite + 本地文本日志 |
| 运行入口 | CLI / 文本控制台 |
| LLM 接口 | 单独客户端模块 |

六、模块依赖方向

初版要尽量避免循环依赖。推荐依赖方向如下：

| 模块 | 可依赖对象 |
|---|---|
| `config` | 无或 utils |
| `world` | core/types |
| `rules` | world、core、events |
| `agents` | config、world、core、events |
| `narrative` | events、world、agents 生成结果 |
| `storage` | world、events、core |
| `interfaces` | app/runtime、narrative、events |
| `app` | 所有模块的装配层 |

依赖原则：

1. `app` 负责装配，不写业务规则。
2. `rules` 不直接依赖 CLI。
3. `agents` 不应直接修改数据库，应通过上层流程写回。
4. `narrative` 只消费事件与状态，不反向改写世界真相。


七、核心运行对象建议

建议初版至少定义以下几个顶层对象：

| 对象 | 职责 |
|---|---|
| `WorldState` | 保存当前世界真实状态 |
| `WorldEngine` | 执行运行循环 |
| `BudgetManager` | 控制 AI 调用预算 |
| `AgentScheduler` | 决定哪些人物被唤醒 |
| `ActionResolver` | 将角色意图裁定为结果 |
| `EventStream` | 追加与管理事件流 |
| `Chronicler` | 生成史官短摘要 |
| `ObserverInterface` | 提供玩家观察入口 |


八、建议的数据流

运行时的主数据流建议如下：

1. `WorldState` 提供当前世界状态。
2. `WorldEngine` 读取状态并推进时间。
3. `rules` 模块更新宏观状态并生成结构候选。
4. `AgentScheduler` 根据候选与预算筛人。
5. `intent_generator` 为少量人物生成结构化意图。
6. `ActionResolver` 解析意图并输出结果。
7. `events.factory` 将结果转为基础事件。
8. `EventStream` 记录并推送基础事件。
9. `Chronicler` 对关键事件生成简短旁白。
10. `storage` 将状态与事件持久化。
11. `interfaces` 将事件流和摘要展示给观察者。


九、配置文件建议

初版建议将可调参数外置配置。

推荐配置文件：

| 文件 | 用途 |
|---|---|
| `config/world.yaml` | 世界初始化参数 |
| `config/runtime.yaml` | 时间推进、事件密度、模式切换参数 |
| `config/ai.yaml` | 模型、预算、prompt 长度、回退策略 |
| `config/narrative.yaml` | 文本风格、摘要长度、观察输出策略 |

建议配置项示例：

| 分类 | 示例项 |
|---|---|
| 世界 | 地区数、文明数、非凡物数、主角数、活跃人物数 |
| 运行 | 默认时间粒度、关键模式阈值、事件流刷新节奏 |
| AI | 主角预算、活跃角色预算、摘要预算、模型名 |
| 文本 | 摘要字数、旁白风格、观察细节密度 |


十、Prompt 组织建议

Prompt 也应模块化存放，避免散落在代码中。

推荐结构：

| 路径 | 用途 |
|---|---|
| `prompts/character_intent.txt` | 角色意图生成模板 |
| `prompts/chronicler_summary.txt` | 史官摘要模板 |
| `prompts/observer_focus.txt` | 聚焦观察模板 |
| `prompts/system_rules.txt` | 固定世界规则和输出约束 |

原则：

1. 固定 prompt 尽量缓存。
2. Prompt 中强调“结构化输出优先”。
3. 不允许角色 prompt 越权创造未授权事实。


十一、存储层建议

初版存储应尽量简单，但结构上要区分不同信息层。

推荐存储对象：

| 存储内容 | 建议方式 |
|---|---|
| 世界状态 | SQLite |
| 事件流 | SQLite + 可选文本日志 |
| 角色短期记忆 | SQLite 或 JSON 缓存 |
| 观察风味缓存 | 本地缓存文件或内存缓存 |
| 配置 | YAML |
| 设计文档 | 文本文件 |

推荐 SQLite 表方向：

| 表名建议 | 内容 |
|---|---|
| `world_state` | 当前世界聚合状态 |
| `civilizations` | 文明数据 |
| `regions` | 地区数据 |
| `characters` | 人物数据 |
| `factions` | 派系数据 |
| `relics` | 非凡物数据 |
| `events` | 结构化事件流 |
| `summaries` | 史官摘要或阶段摘要 |
| `observations` | 玩家观察痕迹 |


十二、CLI 交互建议

初版既然是纯文本，接口层应直接服务于观察者体验。

建议命令集合：

| 命令 | 作用 |
|---|---|
| `run` | 持续运行世界 |
| `step` | 推进一轮或若干轮 |
| `pause` | 暂停运行 |
| `status` | 查看世界总体状态 |
| `watch world` | 查看世界概况 |
| `watch civ <id>` | 观察文明 |
| `watch region <id>` | 观察地区 |
| `watch character <id>` | 观察人物 |
| `watch relic <id>` | 观察非凡物 |
| `events` | 查看近期事件流 |
| `focus <target>` | 提高某对象的观察优先级 |

说明：

1. 初版可只先实现 `run`、`step`、`watch region`、`watch character`、`events`。
2. 命令解析应独立于核心规则。


十三、扩展接口预留建议

你已明确希望后续可扩展，因此骨架中要预留接口。

推荐预留点：

| 扩展点 | 说明 |
|---|---|
| 规则包接口 | 未来可替换“现实未来科技发展风格”的规则集 |
| 模型客户端接口 | 可替换 DeepSeek、GPT 等不同模型 |
| 叙事风格接口 | 可切换不同史官口吻 |
| 存储后端接口 | 未来可从 SQLite 替换为更复杂方案 |
| 观察模式接口 | 后续可加入“观察塑形”模式 |
| 回溯接口 | 后期加入时间线分支时预留位置 |


十四、最小可行实现顺序

建议不要一开始全做完，而是按闭环优先。

推荐实现顺序：

| 顺序 | 内容 |
|---|---|
| 1 | 建立目录结构与配置加载 |
| 2 | 定义世界核心数据模型 |
| 3 | 实现 `WorldState` 与初始化生成 |
| 4 | 实现宏观推进与时间控制 |
| 5 | 实现结构化事件流 |
| 6 | 实现角色候选筛选与预算管理 |
| 7 | 接入 LLM 角色意图生成 |
| 8 | 实现行动裁定 |
| 9 | 实现史官摘要 |
| 10 | 实现 CLI 观察与流式展示 |


十五、初版可以先不做的工程内容

为了收敛项目，以下内容建议初版暂缓：

| 内容 | 原因 |
|---|---|
| Web UI 或桌面 UI | 当前文本界面已足够 |
| 多进程架构 | 初版可以单进程验证闭环 |
| 复杂消息队列 | 会增加工程负担 |
| 向量数据库 | 当前不是刚需 |
| 热更新脚本系统 | 可后置 |
| 分布式运行 | 与当前目标无关 |


十六、代码风格与实现原则

建议从一开始就保持以下原则：

| 原则 | 说明 |
|---|---|
| 数据结构先于文本输出 | 先有对象，再有叙事 |
| 事件是主记录 | 所有历史感从事件流生长 |
| 配置外置 | 重要参数不要硬编码 |
| 模块边界明确 | 世界、规则、AI、叙事、界面分离 |
| 回退路径存在 | AI 失败不能让世界停摆 |
| 文本尽量短 | 初版重连续性，不重长篇文学 |

十八、初版骨架的判断标准

若要判断当前骨架是否合理，可按以下标准检查：

| 检查项 | 标准 |
|---|---|
| 世界能否脱离观察者持续运行 | 必须能 |
| 规则层与 AI 层是否解耦 | 必须尽量解耦 |
| 事件流是否能独立存在 | 必须能 |
| CLI 是否只是读取和触发，不承担核心业务 | 必须是 |
| AI 失败时系统是否可退化运行 | 必须可退化 |
| 后续加规则包或回溯时是否有位置可接 | 必须预留 |
