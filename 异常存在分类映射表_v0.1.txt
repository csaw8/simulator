异常存在分类映射表 v0.1

一、文档目标
本文档用于承接《异常存在层草案_v0.1.txt》，将当前原型中已经存在的 `relic_type` 做一次重新分类。

目标不是立刻改代码，而是先明确：
1. 当前每一种 `relic_type` 在未来异常存在层中应归属到哪里
2. 哪些仍可暂时保留在 `Relic` 壳内
3. 哪些后续应拆成新的异常存在子类
4. 哪些事件与观察语义应随之调整


二、当前原型中的现有类型
当前 `src/world/builder.py` 中定义的类型如下：

| 当前字段 | 当前取值 |
|---|---|
| `_RELIC_TYPES` | `relic_device`, `megastructure`, `sealed_archive`, `founding_protocol` |

这四类已经说明原型内部实际上混入了不止一种存在形态，只是目前都被放在 `Relic` 模型中。


三、重新分类原则
本次映射遵循以下原则：

| 原则 | 说明 |
|---|---|
| 保留兼容性 | 现阶段不要求立刻删掉 `Relic` |
| 优先语义纠偏 | 先确定它“本该是什么” |
| 行为先于命名 | 分类应服务于后续不同事件模板 |
| 面向未来扩展 | 映射结果应能自然连接巨构、异常生命、自主系统等更广层 |


四、第一版映射表
建议的第一版映射如下：

| 当前 `relic_type` | 当前中文理解 | 未来异常存在大类 | 是否继续暂存于 `Relic` 壳 | 后续重点 |
|---|---|---|---|---|
| `relic_device` | 异常装置 / 禁用设备 / 遗留核心 | `relic` 或 `anomalous_object` | 是 | 强化封存、接入、启封、权限争夺 |
| `megastructure` | 巨构 / 工程奇点 / 超大型构造体 | `megastructure` | 否，后续优先拆出 | 强化建造阶段、停摆、并网、崩塌 |
| `sealed_archive` | 封存档案 / 历史黑箱 / 禁区数据库 | `relic` | 是 | 强化解封、信息泄露、文明正统性影响 |
| `founding_protocol` | 创始协议 / 根治理逻辑 / 协议残留 | `autonomous_system` 或 `relic` | 短期是，长期应拆出 | 强化渗透、接管、治理重写、自主演化 |


五、每一类的详细解释

1. `relic_device`
这类对象最接近当前“遗物”直觉。

| 项 | 说明 |
|---|---|
| 存在性质 | 异常物件型 |
| 推荐归类 | `relic` 或 `anomalous_object` |
| 为什么 | 它通常具有可被封存、启用、争夺、调用的装置属性 |
| 是否适合继续叫遗物 | 大体适合 |
| 当前系统是否能兼容 | 能 |

后续建议的典型事件：
- `relic_contest`
- `relic_control`
- `relic_activation`
- `relic_lockdown`
- `relic_access_shift`


2. `megastructure`
这类对象不应长期被视作“物品”，因为巨构不是手持或仓储意义上的对象，而是一种区域级、文明级存在。

| 项 | 说明 |
|---|---|
| 存在性质 | 超大型非人构造存在 |
| 推荐归类 | `megastructure` |
| 为什么 | 它天然带有阶段推进、建设中断、功能恢复、区域重构等行为 |
| 是否适合继续叫遗物 | 不适合，只能短期兼容 |
| 当前系统是否能勉强承接 | 能，但表达严重不足 |

后续建议的典型事件：
- `megastructure_phase_advance`
- `megastructure_stall`
- `megastructure_restart`
- `megastructure_breach`
- `megastructure_failure`

后续建议的典型关系：
- `under_construction_in`
- `stabilizes`
- `destabilizes`
- `controlled_by`
- `depends_on`


3. `sealed_archive`
这类对象更像“历史性异常残留”，仍然适合归入遗物分支，但其价值不在物理力量，而在知识与合法性。

| 项 | 说明 |
|---|---|
| 存在性质 | 历史信息型异常残留 |
| 推荐归类 | `relic` |
| 为什么 | 它更像被封存的文明断片，而不是活跃系统 |
| 是否适合继续叫遗物 | 适合 |
| 当前系统是否能兼容 | 能 |

后续建议的典型事件：
- `archive_breach`
- `archive_unseal`
- `archive_claim`
- `archive_suppression`
- `archive_legitimacy_shock`

它的关键作用不一定是制造物理灾害，而可能是：
- 改变文明合法性
- 颠覆派系叙事
- 改写角色目标


4. `founding_protocol`
这类对象最危险的地方在于它会诱导系统误把“协议”当成“物件”。实际上它更接近一种能持续作用的非人系统性存在。

| 项 | 说明 |
|---|---|
| 存在性质 | 协议型、自主系统型异常存在 |
| 推荐归类 | `autonomous_system`，次选 `relic` |
| 为什么 | 创始协议可能渗透治理、重写权限、诱发制度级连锁 |
| 是否适合长期叫遗物 | 不适合 |
| 当前系统是否能勉强兼容 | 能，但语义不准确 |

后续建议的典型事件：
- `protocol_reactivation`
- `system_infiltration`
- `access_doctrine_shift`
- `governance_rewrite`
- `protocol_lockout`

这类对象更像：
- 还在运行的逻辑残骸
- 仍可干预现实的基础协议
- 有可能重新接管社会结构的系统核心


六、建议补充的未来子类
为了避免后续所有新内容继续往 `Relic` 里塞，建议提前留出以下子类位置：

| 未来子类 | 作用 |
|---|---|
| `anomalous_lifeform` | 容纳异常生物、群体性生体存在 |
| `anomalous_ecology` | 容纳会改变地区生态与生产方式的异常环境 |
| `proto_civil_entity` | 容纳具有组织趋势的非人集群 |
| `hybrid_presence` | 容纳难以归类的混合异常存在 |

这样以后遇到：
- 异常生物的诞生
- 自治菌毯的蔓延
- 巨构施工体的自主演化
- 半协议半生态的扩张体

就不会再被强行命名为“遗物”。


七、当前原型的短期兼容建议
为了不破坏现有运行，建议短期内继续保留 `Relic` 模型，但将其视为：

| 视角 | 含义 |
|---|---|
| 工程视角 | 当前的兼容容器 |
| 设计视角 | 未来异常存在层的暂存壳 |
| 语义视角 | 不再默认翻译为“遗物”，而应根据 `relic_type` 理解其性质 |

也就是说，短期内可以这样理解：

| 当前字段 | 临时语义 |
|---|---|
| `Relic` | 异常存在兼容容器 |
| `relic_type=megastructure` | 实际上是巨构型异常存在 |
| `relic_type=founding_protocol` | 实际上是自主系统型异常存在 |


八、观察层的命名修正建议
后续即使代码暂时不改，观察文本和设计语言也应开始松动。

当前容易误导的说法：
- 遗物
- 传奇物品

后续更合理的内部设计语言：

| 场景 | 推荐说法 |
|---|---|
| `relic_device` | 异常装置 / 封存装置 / 残留核心 |
| `megastructure` | 巨构 / 巨构体 / 超大型构造存在 |
| `sealed_archive` | 封存档案 / 历史黑箱 |
| `founding_protocol` | 创始协议 / 根治理协议 / 协议残留系统 |

这一步不一定先改 CLI 名称，但设计文档和后续实现不应再把它们一概视作“遗物感物件”。


九、对当前代码实现的启示
这一版映射表会直接影响后续代码顺序。

建议的下一步实现优先级：

| 优先级 | 方向 | 原因 |
|---|---|---|
| 高 | 给 `megastructure` 独立事件模板 | 它最不该继续被当普通遗物 |
| 高 | 给 `founding_protocol` 独立事件模板 | 它最像系统性异常存在 |
| 中 | 给 `sealed_archive` 强化信息/合法性类影响 | 与当前物理争夺逻辑区别开 |
| 中 | 给 `relic_device` 保留当前争夺逻辑并细化 | 它最适合当前遗物分支 |


十、结论
当前原型中的四个 `relic_type` 并不属于同一种存在类别。

它们本质上至少跨越了三类未来对象：

| 未来大类 | 当前对应类型 |
|---|---|
| `relic` | `relic_device`, `sealed_archive` |
| `megastructure` | `megastructure` |
| `autonomous_system` | `founding_protocol` |

因此，后续项目的正确方向不是继续扩充“遗物种类”，而是：

1. 承认当前 `Relic` 是兼容壳
2. 用这张映射表指导事件、观察、关系的逐步拆分
3. 为未来“异常存在层”的正式独立做准备


十一、建议的下一步
按照前一份草案的顺序，完成本映射表后，最自然的后续是：

| 顺序 | 下一步 |
|---|---|
| 2 | 给现有 `relic_type` 扩成更明确的异常存在子类语义 |
| 3 | 先从 `megastructure` 和 `founding_protocol` 开始补独立事件模板 |

也就是说，后续不需要一下重构全部，只要先把最不该“遗物化”的两类拆开即可。
