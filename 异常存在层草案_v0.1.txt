异常存在层草案 v0.1

一、文档目标
本文档用于定义项目后续的“异常存在层”。
这里的异常存在层，不是“特殊物品列表”，也不是“遗物系统扩展版”，而是世界中所有重要的、非人的、具有异常性或超常性的存在集合。

它应当能够承载以下内容：
1. 巨构的建造、停摆、再启动、崩塌
2. 异常生物的诞生、扩散、适应、失控
3. 异常生态的形成、蔓延、侵蚀、驯化
4. 自主系统的苏醒、渗透、接管、自复制
5. 历史性异常残留，即当前原型中的“遗物”

本文档目标不是立即替换当前代码中的 `Relic`，而是为后续的结构演进提供一条清晰、可渐进迁移的路线。


二、为什么需要异常存在层
当前原型已经证明：
1. 世界可以围绕非人物对象形成事件链
2. 派系和角色可以围绕这些对象发生争夺、封控、接入、稳定等行为
3. 观察层可以把这些对象纳入长期历史流

但当前实现存在一个明显问题：

| 问题 | 表现 |
|---|---|
| 语义过窄 | 巨构、封存装置、协议核心、异常系统都被压成 `Relic` |
| 存在形式过扁 | 所有重要非人异常对象都像“物品” |
| 演化逻辑混淆 | 巨构、异常生命、异常生态、本应拥有不同的生成与扩散方式，却共用一套争夺逻辑 |
| 世界质感受限 | 世界容易变成“围绕神秘物件争夺”，而不是“人类文明与非人存在共同演化” |

因此，后续必须把“异常存在”从“遗物”中抽出来，成为独立世界层。


三、异常存在层的定义
异常存在层指：
在世界中具有持续存在性、非人物属性、异常性或超常性，并能对地区、组织、角色、文明演化产生结构性影响的对象集合。

这个定义强调四点：

| 维度 | 含义 |
|---|---|
| 非人物 | 不是角色，不以人格化行动为基础 |
| 持续存在 | 不是一闪而过的事件，而是可被长期追踪 |
| 异常性 | 具备超出普通设施或普通资源节点的存在方式 |
| 结构性影响 | 能改变地区秩序、组织策略、角色关注点、文明路径 |


四、异常存在层不是什么
为了避免后续结构继续混乱，需要明确边界。

异常存在层不等于：

| 非目标 | 说明 |
|---|---|
| 普通关键设施 | 例如普通水厂、运输站、仓储节点，应属于设施层 |
| 普通战略节点 | 例如普通关口、物流中枢、行政主机，应属于战略资产层 |
| 普通物品系统 | 武器、载具、工具、样本，不应默认进入该层 |
| 普通事件 | 暂时出现后立即消失的异常现象，不应自动变成长期存在 |

换句话说：
“重要”不等于“异常存在”。
异常存在的重点不是重要，而是其存在方式和演化方式具有异常性。


五、建议的上位结构
后续建议将当前 `Relic` 提升为更广的上位层，暂命名为：

- `ExceptionalPresence`
或
- `Anomaly`
或中文语义层：
- `异常存在`

在 v0.1 草案中，建议统一使用“异常存在”作为设计术语。

其顶层结构建议如下：

| 字段 | 含义 |
|---|---|
| `presence_id` | 异常存在唯一标识 |
| `presence_type` | 存在大类 |
| `subtype` | 细分类 |
| `region_refs` | 当前分布地区，可为一个或多个 |
| `holder_ref` | 若存在明确控制方，则记录；若没有，可为空 |
| `significance` | 叙事与结构重要性 |
| `danger` | 风险等级 |
| `activity_state` | 活跃、沉寂、扩张、封印、失控等 |
| `origin_hint` | 来源线索，例如远古、实验、生态突变、自主演化 |
| `story_tags` | 用于叙事和观察摘要 |
| `linked_events` | 关联历史事件 |
| `interaction_profile` | 该异常存在允许的典型交互方式 |


六、建议的异常存在大类
后续建议至少拆成以下几大类：

| 大类 | 说明 | 示例 |
|---|---|---|
| `relic` | 历史性异常残留 | 封存协议、创始档案、禁用装置 |
| `megastructure` | 巨构类存在 | 轨道环骨架、深井阵列、气候塔、海上城芯 |
| `anomalous_lifeform` | 异常生命体 | 合成群落、变异海兽、群体意识体 |
| `anomalous_ecology` | 异常生态体 | 算法菌毯、放射繁殖林、发光废海 |
| `autonomous_system` | 自主异常系统 | 失控防卫网、自复制维护群、自治工业链 |
| `proto_civil_entity` | 准文明非人实体 | 蜂巢网络、轨道孢群、海底智能群 |
| `hybrid_presence` | 混合存在 | 半基础设施半生命、半协议半生态的复杂体 |

当前原型中的 `Relic`，建议只保留为其中一个子类，而不再承载全部语义。


七、不同异常存在的行为差异
不能只改名，不改行为。
不同子类必须逐渐拥有不同的演化模板。

| 类型 | 关键行为 |
|---|---|
| `relic` | 封存、启封、争夺、调用、转移、权限接管 |
| `megastructure` | 建造、扩展、停摆、重启、崩塌、并网 |
| `anomalous_lifeform` | 诞生、繁殖、迁徙、寄生、适应、突变 |
| `anomalous_ecology` | 扩散、侵蚀、同化、覆盖、驯化、枯竭 |
| `autonomous_system` | 接管、渗透、自修复、自复制、协议扩张 |
| `proto_civil_entity` | 聚合、组织化、学习、模仿、边界扩展 |
| `hybrid_presence` | 组合式行为，根据主导属性决定 |

这意味着后续世界将不仅有“争夺某物”，还会出现：
- 巨构正在被建成
- 异常生物正在形成新群落
- 异常生态正在改变地区生产方式
- 自主系统正在渗透文明的基础治理


八、与当前世界层的关系
异常存在层不是孤立系统，而应与现有层形成双向作用。

| 层级 | 关系方式 |
|---|---|
| 地区层 | 异常存在影响安全、稀缺、生态压力、政治张力 |
| 派系层 | 派系争夺、控制、封锁、利用、研究异常存在 |
| 角色层 | 角色介入封控、接入、研究、猎杀、协商、逃离 |
| 文明层 | 文明围绕异常存在调整发展路径、合法性、扩张模式 |
| 关系层 | 异常存在成为关系网络节点，不只是被观察对象 |

简化理解：
异常存在层应成为“非人历史的驱动力”，而不是“被争夺的道具层”。


九、与当前 `Relic` 的迁移关系
为了不破坏现有原型，建议渐进迁移，而不是一次推翻。

建议路线如下：

| 阶段 | 做法 |
|---|---|
| 第一阶段 | 重新定义 `Relic` 语义：它只是异常存在的一种具体子类 |
| 第二阶段 | 新增上位概念，如 `ExceptionalPresence` |
| 第三阶段 | 让 `relic_type` 中原有值逐步映射到新的子类 |
| 第四阶段 | 新事件、新关系、新观察优先面向上位层编写 |
| 第五阶段 | 原 `Relic` 作为兼容壳逐步收窄，只保留历史性异常对象 |

当前已有 `relic_type` 可作如下初步映射：

| 当前类型 | 未来建议映射 |
|---|---|
| `relic_device` | `relic` / `anomalous_object` |
| `megastructure` | `megastructure` |
| `sealed_archive` | `relic` |
| `founding_protocol` | `autonomous_system` 或 `relic` |


十、异常存在层的事件模板建议
后续事件不应都叫“争夺遗物”。
建议按类型逐步拆分。

| 类别 | 示例事件类型 |
|---|---|
| 通用异常事件 | `presence_emergence`, `presence_dormancy_shift`, `presence_destabilization` |
| 遗物事件 | `relic_contest`, `relic_control`, `relic_access_lock`, `relic_activation` |
| 巨构事件 | `megastructure_phase_advance`, `megastructure_stall`, `megastructure_restart`, `megastructure_failure` |
| 异常生命事件 | `lifeform_birth`, `lifeform_spread`, `lifeform_adaptation`, `lifeform_breach` |
| 异常生态事件 | `ecology_spread`, `ecology_consumption`, `ecology_mutation`, `ecology_containment` |
| 自主系统事件 | `system_infiltration`, `system_takeover`, `system_replication`, `system_lockdown` |

这样世界将开始出现真正不同的非人历史线。


十一、异常存在层的关系模板建议
既然当前项目已经开始有动态关系层，异常存在必须天然进入关系网络。

建议优先支持以下关系：

| 关系类型 | 含义 |
|---|---|
| `anchored_in` | 某异常存在锚定在某地区 |
| `spreading_into` | 某异常存在向某地区扩散 |
| `contested_by` | 某异常存在正被某派系或角色争夺 |
| `controlled_by` | 某异常存在当前受某组织控制 |
| `contained_by` | 某异常存在被某角色或组织压制 |
| `studied_by` | 某异常存在被研究中 |
| `depends_on` | 某地区或组织依赖某异常存在的副产物 |
| `distorted_by` | 某地区被异常存在持续扭曲 |
| `coexists_with` | 某异常生命或生态与某地区形成脆弱共存 |

这些关系会让异常存在从“事件触发器”变成“关系节点”。


十二、观察系统如何接入
异常存在层接入后，观察系统应从四个角度描述它：

| 观察维度 | 说明 |
|---|---|
| 客观状态 | 类型、地区、状态、危险度、控制方 |
| 关系状态 | 正被谁争夺、谁依赖它、谁在压制它 |
| 演化状态 | 正在建造、扩散、沉寂、繁殖、崩塌还是并网 |
| 叙事状态 | 它如何改变地区气氛、组织决策、角色关注点 |

因此后续可扩出的观察命令包括：
- `watch presence <id> brief|full`
- 或保留旧命令兼容，但底层统一归入异常存在层


十三、初期实现建议
为了与当前原型平滑衔接，建议不要一步做满，而是分三步走。

第一步：语义纠偏
1. 在设计上确认 `Relic` 不是总类
2. 新增上位概念“异常存在”
3. 新生成内容开始按子类思考

第二步：行为拆分
1. 在当前 `Relic` 壳内，先扩出更细的行为模板
2. 至少区分：
   - 巨构型事件
   - 遗物型事件
   - 自主系统型事件
3. 观察摘要和事件流开始体现差异

第三步：结构迁移
1. 单独抽出新的异常存在模型
2. 让旧 `Relic` 向上位层兼容迁移
3. 让关系层、事件层、观察层统一面向上位层


十四、与项目风格的适配
用户已确定世界风格是“现实未来科技发展风格”。
因此异常存在层不应默认走纯神秘奇幻路线，而应优先满足以下质感：

| 风格原则 | 说明 |
|---|---|
| 技术性 | 异常存在常与科技、工业、协议、生态工程相关 |
| 现实性 | 它们带来的影响应落实到供给、秩序、治理、迁移、风险 |
| 非游戏化 | 不追求严格数值平衡，而强调结构性后果和故事连续性 |
| 长时演化 | 异常存在应像天气、物种、制度一样逐步演变，而不是只在观察时发生 |


十五、结论
当前项目后续最重要的修正之一，不是“让遗物少一点”，而是：

把“遗物”从一个过于狭窄又过于泛化的名词中解放出来，
将其放回“异常存在层”的一个子类位置。

最终理想状态不是：
- 世界里有一堆重要遗物

而是：
- 世界里有一套非人异常存在谱系
- 其中遗物只是谱系中的一支
- 巨构、异常生物、异常生态、自主系统等都能独立演化
- 文明、地区、派系、角色与这些存在共同构成历史流


十六、建议的下一步
如果基于当前原型继续推进，最合理的后续顺序建议如下：

| 顺序 | 建议 |
|---|---|
| 1 | 先保留现有 `Relic` 实现，但重新定义其设计语义 |
| 2 | 给现有 `relic_type` 扩成更明确的异常存在子类 |
| 3 | 给不同子类补不同事件模板 |
| 4 | 让观察系统开始使用“异常存在”而不是只说“遗物” |
| 5 | 再决定是否正式引入新的上位模型类 |

这条路线能够兼顾：
1. 当前原型的连续可运行性
2. 世界风格的统一性
3. 后续自由度提升的空间
