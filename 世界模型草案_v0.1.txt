世界模型草案 v0.1

一、文档目标

本文档用于定义项目初版的世界模型边界，为后续的运行循环、AI 调度、事件流展示与存储结构提供统一基础。

当前版本以以下原则为前提：

1. 项目是“文本叙事世界引擎”，不是传统高精度数值游戏。
2. 保留宏观层面的客观性，但不追求严格数值模拟。
3. 世界以连续事件流运行，玩家主要作为观察者。
4. 宏观世界常驻运行，局部人物与细节按重要性分层。
5. AI 重点驱动少数主角与部分活跃人物，大量普通人物不进行持续个体模拟。
6. 初版不做回溯与时间线分支。


二、世界模型总览

世界采用“三层分辨率 + 一条连续事件流”的结构。

| 层级 | 主要对象 | 运行方式 | 持久性 | AI 参与度 |
|---|---|---|---|---|
| 宏观层 | 文明、地区、组织格局、科技阶段、社会压力、战争态势 | 本地规则推进 | 强持久 | 低 |
| 中观层 | 主角、活跃人物、重要组织节点、传奇物品、巨构 | 规则裁定 + AI 意图驱动 | 中到强持久 | 高 |
| 微观层 | 普通人物、普通物件、局部景象、传闻、街道细节 | 观察时生成 | 弱持久或不持久 | 按需 |
| 事件流 | 所有已发生且被系统记录的结构化事件 | 按时间顺序持续追加 | 强持久 | 仅叙事层参与 |


三、世界的最小核心对象

初版世界由以下六类核心对象组成：

| 对象类型 | 是否必须 | 作用 |
|---|---|---|
| Civilization 文明 | 是 | 世界的宏观历史主体 |
| Region 地区 | 是 | 承载局势变化、资源压力和具体事件 |
| Character 人物 | 是 | 世界的行动主体，分层驱动 |
| Faction 派系/组织 | 是 | 连接文明、地区与人物关系 |
| Relic 非凡物 | 是 | 承载权力、象征、危机与文明遗产 |
| Event 事件 | 是 | 世界演化的唯一连续主记录形式 |


四、文明 Civilization 模型

文明是宏观层主体，用于承载长期历史趋势，而不是精细管理每个个体。

建议字段如下：

| 字段 | 类型建议 | 含义 |
|---|---|---|
| civ_id | string | 文明唯一 ID |
| name | string | 文明名称 |
| origin_region_id | string | 起源地区 |
| status | enum | 存续、扩张、收缩、裂解、残响、消亡 |
| stage | enum | 当前文明阶段 |
| trajectory | enum/list | 当前发展路径倾向 |
| governance_mode | enum | 统治形态 |
| cohesion | tier + trend | 内部凝聚度 |
| expansion_pressure | tier + trend | 扩张冲动 |
| scarcity_pressure | tier + trend | 生存与资源压力 |
| tech_profile | enum/list | 技术轮廓 |
| belief_profile | enum/list | 信念与意识形态轮廓 |
| military_posture | tier | 军事姿态 |
| legitimacy | tier + trend | 正统性/合法性 |
| external_relations | refs | 与其他文明关系摘要 |
| key_regions | list | 关键地区列表 |
| key_factions | list | 文明内重要组织 |
| key_characters | list | 当前被重点关注的人物 |
| key_relics | list | 当前文明持有的重要非凡物 |
| summary_tags | list | 用于压缩描述的标签 |

说明：

1. 文明字段尽量使用“层级/标签/趋势”，不使用高精度资源数字。
2. 文明不是地图染色块，而是一个历史结构体。
3. 一个文明可以控制多个地区，也可能在裂解后变成多个次级政治体。


五、文明阶段 Stage 模型

文明阶段用于提供历史感、发展节奏与事件权重倾向。初版采用固定通用模板。

建议初始阶段集：

| 阶段 | 含义 | 常见压力 | 常见机会 |
|---|---|---|---|
| 拓荒期 | 新秩序尚未成形，基础设施稀薄 | 生存、迁徙、掠夺、环境风险 | 资源发现、定居、原始联盟 |
| 聚落期 | 核心据点稳定，组织与习俗初步形成 | 内部分工不足、边界脆弱 | 城镇成长、制度萌芽 |
| 城邦/诸域期 | 多地区并立竞争，地方认同增强 | 战争、外交失衡、地方割裂 | 联盟、兼并、文化扩散 |
| 联邦/帝国期 | 高度整合，出现跨区域秩序与标准化 | 官僚臃肿、边疆过载、异端积累 | 大工程、秩序输出、技术整合 |
| 高技跃迁期 | 科技体系成为主要组织力量 | 失控创新、治理滞后、阶层断裂 | 产能飞跃、远程统治、认知革命 |
| 裂解期 | 中心失效，区域与组织重新争夺秩序 | 叛乱、饥荒、信任崩塌 | 新秩序萌芽、英雄崛起 |
| 残响期 | 原文明失去主体连续性，只剩制度、遗迹与传说 | 被掠夺、被神话化、遗产争夺 | 遗产继承、复兴、异化重组 |

说明：

1. 你的世界风格是“现实未来科技发展风格”，所以“高技跃迁期”应为强阶段。
2. 阶段不强制线性推进，允许从高技跃迁期直接跌入裂解期。
3. 阶段切换通常由多个结构性事件叠加触发，不应由单条偶发事件直接决定。


六、文明发展路径 Trajectory 模型

发展路径不是阶段，而是文明长期偏向。一个文明可以同时有主路径与副路径。

建议路径集：

| 路径 | 含义 |
|---|---|
| 平台扩张 | 依靠基础设施、交通、能源网络扩展统治 |
| 技术统合 | 依靠科研与系统标准化形成秩序 |
| 企业寡头 | 资本实体或超级企业主导社会结构 |
| 军政稳定 | 依靠军队、安全系统与边疆控制维持秩序 |
| 算法治理 | 决策、资源与身份被计算系统强约束 |
| 信念动员 | 意识形态、宗教或共同叙事维系社会 |
| 碎片寄生 | 依靠旧文明残骸、遗产设施和废墟资源延续 |
| 生态调适 | 文明重点变为环境适配与生存工程 |

建议实现方式：

1. 每个文明至少有一个主路径。
2. 路径影响事件权重，而不是直接决定所有结果。
3. 路径可随阶段变化而偏移，但不应频繁抖动。


七、地区 Region 模型

地区是初版最重要的演化单元之一。它比人物更稳定，比文明更具体。

建议字段如下：

| 字段 | 类型建议 | 含义 |
|---|---|---|
| region_id | string | 地区唯一 ID |
| name | string | 地区名称 |
| civ_id | string/null | 当前所属或主导文明 |
| region_type | enum | 都市、工业带、边疆、港区、轨道区、荒野、矿区、科研区等 |
| connectivity | tier | 与其他地区的连接性 |
| security | tier + trend | 治安与秩序状况 |
| prosperity | tier + trend | 繁荣度 |
| scarcity | tier + trend | 资源紧张度 |
| infrastructure | tier + trend | 基础设施状态 |
| tech_density | tier | 技术密度 |
| ecological_stress | tier + trend | 环境压力 |
| political_tension | tier + trend | 政治紧张度 |
| belief_temperature | tier | 信念/意识形态温度 |
| population_profile | tags | 人口轮廓，不要求精确数值 |
| strategic_value | tier | 战略重要度 |
| active_factions | list | 当前活跃组织 |
| active_characters | list | 当前被地区系统关注的人物 |
| resident_relics | list | 当前存在的重要非凡物 |
| local_story_hooks | list | 便于 AI 补完的局部标签 |

说明：

1. 初版建议地区数量控制在 8-15 个，与你已确认的规模一致。
2. 地区是世界自动推进的主战场。
3. 大多数结构性事件应当先在地区层出现，再上升为文明级影响。


八、派系/组织 Faction 模型

派系是中观层的稳定结构，用来避免所有变化都压到个人身上。

建议字段如下：

| 字段 | 类型建议 | 含义 |
|---|---|---|
| faction_id | string | 派系唯一 ID |
| name | string | 派系名称 |
| faction_type | enum | 政府、企业、军团、工会、科研机构、教团、地下网络等 |
| parent_civ_id | string/null | 主要隶属文明 |
| power_scope | enum | 地区级、跨地区级、文明级 |
| influence | tier + trend | 影响力 |
| cohesion | tier | 组织内凝聚力 |
| doctrine_tags | list | 行动逻辑与价值倾向 |
| controlled_regions | list | 实际控制地区 |
| key_characters | list | 关键人物 |
| rival_factions | list | 对手组织 |
| allied_factions | list | 盟友组织 |

说明：

1. 很多事件应被解释为“人物 + 派系 + 地区压力”的共同产物。
2. 派系能显著降低对大量个体 AI 的依赖。


九、人物 Character 模型

人物是世界叙事的主要行动体，但不是所有人物都要同精度运行。

初版人物分四级：

| 级别 | 名称 | 数量建议 | 驱动方式 | 持久性 |
|---|---|---|---|---|
| L0 | 背景人口 | 无限近似 | 不单独模拟 | 无个体持久 |
| L1 | 配角人物 | 可较多 | 模板/条件生成 | 弱持久 |
| L2 | 活跃人物 | 20-40 | 条件唤醒 AI | 中持久 |
| L3 | 主角人物 | 5-8 | 高频 AI 跟进 | 强持久 |

人物通用字段建议如下：

| 字段 | 类型建议 | 含义 |
|---|---|---|
| char_id | string | 人物唯一 ID |
| name | string | 人物名称 |
| character_level | enum | L0-L3 |
| current_region_id | string | 当前所在地区 |
| affiliation | refs | 所属派系/文明/网络 |
| role_tags | list | 身份标签 |
| capability_tags | list | 能力标签 |
| desire_tags | list | 欲望/长期倾向 |
| fear_tags | list | 恐惧/回避倾向 |
| loyalty_map | refs/tags | 忠诚对象与偏向 |
| relationship_refs | refs | 关键关系对象 |
| status | enum | 存活、受伤、失势、逃亡、失踪、囚禁、死亡等 |
| notoriety | tier | 声名/关注度 |
| initiative | tier | 主动性 |
| memory_summary | text short | 压缩长期记忆摘要 |
| recent_goal | text short | 最近目标 |
| last_intent | structured | 上次 AI 产出的意图对象 |
| wake_priority_seed | numeric/tier | 唤醒优先级基础值 |
| observation_trace | list/tier | 玩家是否观察过及次数 |

说明：

1. 主角人物应是少数真正长期跟进的对象。
2. 活跃人物是事件引擎和 AI 调度的主池。
3. 配角人物可以从群体中析出，也可以在观察时生成。


十、人物类型：反应者与规划者

你已明确希望“大部分人物是配角式反应者，少部分人物是主角式规划者”。这应写入模型。

建议字段：

| 字段 | 取值建议 | 含义 |
|---|---|---|
| agency_mode | reactive / opportunistic / strategic | 行动模式 |

含义如下：

| 模式 | 含义 | 适用对象 |
|---|---|---|
| reactive | 主要对环境压力做反应 | 大多数配角与部分活跃人物 |
| opportunistic | 在局势变化中主动抓机会 | 一部分活跃人物 |
| strategic | 具备长期布局能力 | 主角人物及少数核心角色 |

说明：

1. 只有少数 strategic 人物需要高频 AI 驱动。
2. reactive 人物更多通过地区危机和组织压力被唤醒。


十一、非凡物 Relic 模型

初版不做一般物品系统，只做重要异常对象。

建议对象范围：

| 类别 | 示例 |
|---|---|
| 传奇道具 | 核心设备、遗失武器、权限密钥、神秘终端 |
| 巨构 | 轨道电梯、深层反应堆、封锁墙、星港、古老数据井 |
| 圣物/禁物 | 黑盒、禁忌数据库、意识祭坛、冷冻样本库 |
| 制度遗产 | 宪章、主协议、创始条约、被遗忘的监管授权 |

建议字段如下：

| 字段 | 类型建议 | 含义 |
|---|---|---|
| relic_id | string | 唯一 ID |
| name | string | 名称 |
| relic_type | enum | 道具、巨构、圣物、制度遗产等 |
| current_region_id | string | 所在地区 |
| holder_ref | char/faction/region | 当前持有或控制者 |
| significance | tier | 历史重要度 |
| danger | tier | 风险等级 |
| activation_state | enum | 休眠、争夺中、激活、失控、封存 |
| story_tags | list | 叙事标签 |
| linked_events | list | 关联重大事件 |

说明：

1. 非凡物不是装备系统，而是事件触发器与历史锚点。
2. 它们应当稀缺，并能跨角色、跨地区留下影响。


十二、宏观状态模型：模糊客观性

初版不做高精度数字经济，而做“模糊但稳定”的客观状态。

推荐状态表达方式：

| 状态维度 | 表达方式 |
|---|---|
| 当前档位 | 低 / 中 / 高 或 充裕 / 紧张 / 失控 |
| 趋势 | 上升 / 持平 / 下降 |
| 波动因子 | 近期扰动标签 |

建议宏观状态集：

| 状态 | 适用层级 | 说明 |
|---|---|---|
| scarcity | 文明/地区 | 资源与补给压力 |
| stability | 文明/地区 | 秩序与治理稳定度 |
| expansion_pressure | 文明 | 向外扩张的冲动与需求 |
| fracture_risk | 文明/地区 | 裂解与分离风险 |
| innovation_pressure | 文明/地区 | 技术变革张力 |
| infrastructure_strain | 地区 | 设施过载或衰退 |
| ideological_heat | 文明/地区 | 意识形态或信念温度 |
| security_tension | 地区 | 武装冲突与安全风险 |
| ecological_stress | 地区 | 环境危机和适居压力 |
| legitimacy | 文明/组织 | 正统性与统治信任 |

说明：

1. 宏观状态推动结构性大事件。
2. 宏观状态不直接写故事，但决定故事发生的概率和方向。


十三、事件 Event 模型

事件是系统最重要的对象。初版一切“历史连续性”都依赖事件流。

建议字段如下：

| 字段 | 类型建议 | 含义 |
|---|---|---|
| event_id | string | 事件唯一 ID |
| tick | integer/time | 发生时间标记 |
| time_granularity | enum | 周、日、半日、小时 |
| event_type | enum | 事件类型 |
| event_scope | enum | 文明级、地区级、角色级、物级 |
| region_refs | list | 相关地区 |
| civ_refs | list | 相关文明 |
| actor_refs | list | 参与人物 |
| faction_refs | list | 参与派系 |
| relic_refs | list | 关联重要物 |
| cause_tags | list | 原因标签 |
| result_tags | list | 结果标签 |
| severity | tier | 事件强度 |
| novelty | tier | 新颖度 |
| consequence_score | tier | 长期影响潜力 |
| narrative_priority | tier | 是否值得进入叙事层 |
| visibility | enum | 明显、半隐藏、隐藏 |
| summary_struct | short object | 结构化摘要 |
| chronicle_text | text/null | 史官文本，非必须当场生成 |

说明：

1. 事件流是世界主记录，不是附属日志。
2. 文本叙事是事件的投影，而不是事件本身。
3. 连续事件链应比单个“大新闻”更重要。


十四、事件分类建议

初版建议至少支持以下事件类别：

| 事件类别 | 示例 |
|---|---|
| 结构性压力事件 | 供给紧张、迁徙、基础设施崩坏、治安失控 |
| 政治事件 | 任命、弹劾、政变、分裂、结盟 |
| 军事事件 | 冲突升级、边疆推进、围困、撤退、镇压 |
| 技术事件 | 试验成功、系统失控、设备复苏、协议泄露 |
| 信念事件 | 宣言、异端传播、神话化、群众狂热 |
| 社会事件 | 罢工、暴乱、逃亡潮、灾后互助 |
| 角色事件 | 背叛、刺杀、招募、失踪、告白、逃亡 |
| 非凡物事件 | 启封、夺取、激活、失控、献祭、封存 |
| 阶段事件 | 文明晋升、衰退、裂解、复兴 |


十五、连续事件流模型

你已明确世界应当“边运行边流式看到事件出现”，因此事件流要具备以下特性：

| 特性 | 说明 |
|---|---|
| 连续性 | 事件按时间顺序持续追加 |
| 因果性 | 事件之间要能追溯前因 |
| 粒度可变 | 平稳期以周为主，关键期可切到日或更细 |
| 可压缩 | 大量基础事件可被摘要成阶段性旁白 |
| 可聚焦 | 玩家可以按地区、人物、文明筛选事件流 |

建议实现概念：

1. 基础事件流：规则与裁定层生成的结构化事件。
2. 叙事事件流：从基础事件流中挑选高价值事件生成短文本。
3. 观察事件流：玩家聚焦某地区或人物时生成的局部观察记录。


十六、时间模型

初版默认时间单位为“周”，但应允许自动细化。

建议时间粒度规则如下：

| 世界状态 | 默认粒度 |
|---|---|
| 平稳时期 | 周 |
| 地区紧张上升 | 日 |
| 战争、政变、灾变、重大事故 | 半日或小时 |
| 玩家主动聚焦 | 临时细化 |

说明：

1. 时间细化应由事件密度触发，而不是玩家手动管理为主。
2. 细化后的结果仍应最终折叠回统一事件流。


十七、角色唤醒模型

AI 成本控制的关键在于：不是所有活跃人物都持续思考。

建议唤醒条件如下：

| 条件 | 说明 |
|---|---|
| 所在地区危机升级 | 环境压力推高 reactive 人物出场概率 |
| 关键关系对象出事 | 触发人物连锁反应 |
| 自身目标接近实现/破灭 | 提高主动行动可能 |
| 所属派系进入冲突 | 组织级变化推动人物唤醒 |
| 玩家近期观察过 | 提高优先级 |
| 已经是主角型人物 | 保持较高唤醒频率 |
| 可能升格 | 配角临近重要事件时进入候选池 |

建议唤醒优先级字段：

| 字段 | 含义 |
|---|---|
| base_priority | 角色级别与身份自带的重要度 |
| tension_bonus | 地区/组织危机带来的加成 |
| relation_bonus | 关键关系对象变化加成 |
| observer_bonus | 玩家关注加成 |
| narrative_bonus | 当前事件链中的戏剧潜力 |


十八、观察生成模型

你已确认初版观察模式为“纯读取”。因此观察不会改写世界事实，只会揭示已有上下文的具体投影。

建议观察目标：

| 目标 | 结果 |
|---|---|
| 观察文明 | 得到阶段、压力、主要矛盾、近期关键事件 |
| 观察地区 | 得到地区局势、组织分布、风险、近期流言或氛围 |
| 观察人物 | 得到该人物状态、关系、近期目标、行为倾向 |
| 观察非凡物 | 得到其来历、争夺现状、象征与风险 |

观察生成原则：

1. 初版观察只读，不改变世界状态。
2. 观察文本可以是一次性风味信息，不要求全部持久化。
3. 对已是主角或关键对象的观察可记录“观察痕迹”，用于后续唤醒加权。


十九、生成与持久化分层

为了避免信息污染，世界信息必须分层存储。

| 层级 | 内容 | 存储策略 |
|---|---|---|
| State | 客观运行状态 | 持久化 |
| Canon | 已确认正史的重大事件、主角、非凡物记录 | 持久化 |
| Working Memory | 活跃角色近期意图、局势摘要、事件缓存 | 中短期持久 |
| Flavor | 观察时生成的局部描写、一次性细节 | 可丢弃或短缓存 |

说明：

1. 不是所有生成文本都应写入正史。
2. 世界真状态与叙事风味必须分离。


二十、初版世界推荐规模

根据你已确认的参数，初版规模建议如下：

| 维度 | 推荐值 |
|---|---|
| 地区数量 | 8-15 |
| 文明数量 | 2-5 个主要文明/政治体 |
| 主角人物 | 5-8 |
| 活跃人物 | 20-40 |
| 派系数量 | 8-20 |
| 非凡物数量 | 5-15 |
| 默认时间单位 | 周 |

说明：

1. 这个规模足以形成连续历史感。
2. 同时仍适合文本驱动和条件唤醒式 AI 调度。


二十一、初版不做的内容

为了收敛实现，以下内容建议明确排除：

| 内容 | 原因 |
|---|---|
| 回溯/分支时间线 | 会显著抬高状态管理复杂度 |
| 一般物品经济系统 | 对叙事收益低，复杂度高 |
| 全人口个体模拟 | 成本和噪声过高 |
| 精细空间路径/地图寻路 | 初版用地区连接关系即可 |
| 玩家主动塑形观察 | 初版先保持只读 |
| 高精度数值平衡系统 | 与项目目标不符 |


二十二、初版世界模型的核心判断

若后续实现过程中出现分歧，应优先遵循以下判断顺序：

| 优先级 | 判断准则 |
|---|---|
| 1 | 是否能保证世界在无人观察时也持续运行 |
| 2 | 是否能保证宏观事件具备客观来源 |
| 3 | 是否能让少量主角和部分活跃人物持续产出故事 |
| 4 | 是否避免为大量不重要对象支付高 AI 成本 |
| 5 | 是否允许玩家通过文本连续观察历史形成过程 |
| 6 | 是否为后续扩展规则包、世界风格和新系统保留接口 |
