Terrarium Project 第三版任务清单与完成顺序

日期：2026-06-12

一、版本判断

| 项 | 结论 |
|---|---|
| 第一版 | 固定结构可运行原型，已完成 |
| 第二版 | AI 深接入前的固定模型建设与收口，已完成 |
| 当前是否进入第三版 | 是，已进入第三版前期 |
| 第三版名称 | AI 动态结构进入世界循环版 |
| 第三版核心目标 | 让 AI 生成的动态结构在受控条件下长期存在、可观察、可审计，并通过 Event / Relation / PressureThread 影响固定世界 |

第三版不是继续给固定对象堆字段，也不是立刻让 AI 自由改世界 schema。

第三版的正确方向是：

```text
固定结构保持稳定
AI 生成 DynamicStructure / 后续半独立实体
所有写入走 proposal -> validation -> audit -> apply
所有影响走 Event / Relation / PressureThread
```

三、第三版总目标

| 目标 | 验收标准 |
|---|---|
| AI 动态结构可安全落库 | `apply=dynamic` 能生成 DynamicStructure / Event / Relation / PressureThread，不污染固定结构 |
| 动态结构有生命周期 | active / cooling / archived 或同等机制，长跑不会无限堆积 |
| 动态结构可去重/合并 | 相同地区、相同类型、相同牵连对象不会重复刷出大量结构 |
| 动态结构进入观察层 | region / faction / relic / project / supply / node 摘要能自然显示相关动态结构 |
| 动态结构进入决策上下文 | intent / fallback / observer context 能看到相关动态结构 |
| AI 写回可追踪 | 每次 AI 提案、拒绝、落库、影响对象都有 audit |
| 可开始半独立新结构 | 在 DynamicStructure 跑稳后，设计 EmergentPresence / AdaptiveLifeform 一类半独立实体 |

四、完成顺序

| 顺序 | 批次 | 任务 | 目标 | 验收 |
|---:|---|---|---|---|
| 1 | V3-A | `apply=dynamic` 安全抽样 | 验证真实 AI proposal 能安全落库 | 1-3 条 live apply 生成结构，固定对象字段不被直接改 |
| 2 | V3-B | 动态结构观察融合 | 让 region/faction/relic/project/supply/node 摘要能提到相关动态结构 | player 不泄漏 `dyn_`，truth 可追踪结构 ID |
| 3 | V3-C | 动态结构生命周期 | 避免 dynamic structures 无限堆积 | 长跑 50-100 tick 后数量稳定，有 archived/cooling 机制 |
| 4 | V3-D | 去重与合并 | 避免 AI 重复生成同类结构 | 同 target/type/linked refs 的重复 proposal 被 update 或 reject |
| 5 | V3-E | 目标选择策略 | 不对所有对象都请求 AI proposal | 只对高 signal 对象建议 proposal，成本可控 |
| 6 | V3-F | 动态结构进入 intent/observer context | 让角色和观察能感知动态结构 | intent prompt/context 包含相关 dynamic structures |
| 7 | V3-G | apply 后长跑验收 | 验证动态结构不会破坏固定世界循环 | 全量测试、compileall、30/100 tick 抽样通过 |
| 8 | V3-H | 半独立实体设计 | 准备新生命存在形式等独立结构 | 输出 EmergentPresence v1 设计，不急于落库 |
| 9 | V3-I | EmergentPresence v1 | 新生命/异常生态第一版 | 独立数据层、proposal、validator、summary、snapshot、测试 |
| 10 | V3-J | SQLite 导出器与基础查询 | 为长期事件、关系、审计和动态结构提供查询层 | 不替代 JSON 快照，可导出 events / relations / dynamic_structures / audits |
| 11 | V3-K | SQLite 增量 ledger 与审计分析 | 为长跑和 AI proposal 质量分析提供持久记录 | 支持 db stats / db audits / db dynamics 等查询 |
| 12 | V3-L | 风格解耦准备 | 为第四阶段 WorldStyleProfile / NarrativeLexicon 做盘点和边界设计 | 输出硬编码清单、默认题材词表、第四阶段接口建议 |
| 13 | V3-M | 隐标签与描述档案层设计 | 为人物性格、建筑外观、物品功能、异常感官特征建立轻量 DescriptorProfile 规划 | 输出 descriptor 字段边界、固定池来源、AI 参与初始化边界 |

六、第三版不建议立即做的事

| 不建议 | 原因 |
|---|---|
| 立刻新增大量 dynamic structure 类型 | 当前五类还没完成生命周期和去重 |
| 让 AI 直接修改固定对象字段 | 污染风险高，难回滚 |
| 自动每步调用 AI 生成结构 | 成本和噪声都会爆炸 |
| 直接做新生命存在固定结构 | DynamicStructure 还未进入稳定世界循环 |
| 批量 `apply=dynamic` | 当前 live 样本还少，应先小规模抽样 |
| 立刻用 SQLite 替代 JSON 快照 | 当前世界模型仍在快速变化，过早数据库化会增加迁移成本 |
| 在第三阶段全面解耦科幻文风 | 会打断 EmergentPresence 主线，且容易稀释当前项目辨识度 |
| 在第三阶段让 AI 自由生成初始化标签 | 会污染固定世界初始状态，应先做固定池、候选权重和审计边界 |

第四阶段不是“去科幻化”，而是把当前科幻异常文明风格集中管理。第三阶段只负责记录债务和预留接口，避免在 V3-H / V3-I 新增结构时继续扩大硬编码范围。

十一、第五阶段预留：半开放结构生成与 AI 模板提案

| 项 | 内容 |
|---|---|
| 第五阶段名称 | 半开放结构生成与 AI 模板提案阶段 |
| 核心任务 | 让 AI 可以提出新的结构模板，但不能直接改 Python schema 或 WorldState 字段 |
| 与第三阶段关系 | 第三阶段只验证 DynamicStructure / EmergentPresence 的受控写入闭环，不开放 schema |
| 与第四阶段关系 | 第四阶段提供 WorldStyleProfile / NarrativeLexicon，作为第五阶段模板生成的题材边界 |
| 推荐实现方式 | AI 提出 `StructureTemplateProposal`，系统 validator + 审批后进入 approved template registry |
| 详细文档 | `项目文档/阶段规划/第五阶段半开放结构生成规划_2026-06-12.txt` |

第五阶段不是让 AI 自由写代码，也不是让 AI 直接成为世界模型 schema 作者。

正确边界是：

```text
AI 可以提出受限 JSON 模板
系统验证模板字段、生命周期、影响通道和查询成本
人类或规则审批后模板才进入可实例化 registry
之后 AI 只能实例化 approved template
固定结构字段仍然不能被 AI 直接改写
```

该阶段应放在第三阶段 DynamicStructure / EmergentPresence 稳定、第四阶段风格边界集中管理之后。这样可以扩大生成自由度，同时避免结构污染、审计断裂和长期存档不可迁移。

十二、V3-M 预留：隐标签与描述档案层

| 项 | 内容 |
|---|---|
| 任务名称 | 隐标签与描述档案层设计 |
| 推荐结构 | `DescriptorProfile` |
| 覆盖对象 | character / region_node / relic / project / supply / dynamic_structure / emergent_presence |
| 覆盖标签 | 人物性格、行为倾向、建筑外观、物品外观、物品功能、异常感官特征、社会观感 |
| 第三阶段目标 | 只做设计与边界，不急于把所有对象重写 |
| 与第四阶段关系 | 第四阶段把 descriptor 词表并入 WorldStyleProfile / NarrativeLexicon |
| 与第五阶段关系 | 第五阶段的半开放结构模板必须声明 descriptor 字段约束 |
| 详细文档 | `项目文档/阶段规划/隐标签与描述档案层规划_2026-06-12.txt` |

推荐轻量结构：

```python
@dataclass(slots=True)
class DescriptorProfile:
    ref_id: str
    profile_type: str  # character / place / object / structure / presence
    appearance_tags: list[str]
    function_tags: list[str]
    behavior_tags: list[str]
    sensory_tags: list[str]
    social_read_tags: list[str]
    notes: list[str]
```

AI 可以参与世界初始化标签生成，但边界应是：

```text
默认：固定池随机抽取，保证离线可复现
可选：AI 只提出候选标签组合或权重调整
必须：候选标签必须来自 approved descriptor pool
必须：validator 校验数量、冲突、题材边界和对象类型适配
必须：AI 初始化结果写入 audit，并可用 seed/fallback 重放
禁止：AI 直接新增固定对象字段或绕过固定池写入任意标签
```

十三、第三阶段最终复核：是否还需要额外改进

本次复核排除已经排入 V3-H 到 V3-M、第四阶段、第五阶段的内容后，第三阶段不建议再扩展新的大系统。

| 复核项 | 判断 | 处理 |
|---|---|---|
| DynamicStructure 安全闭环 | 已具备 | 不再继续扩宽类型，保持五类白名单 |
| proposal / validation / audit / apply | 已具备 | 后续新结构复用该模式 |
| player/truth 防泄漏 | 已有测试覆盖 | 后续随 EmergentPresence 增加同类测试 |
| 长跑稳定性 | 100 tick 抽样稳定 | V3-I 后再做新一轮长跑 |
| AI 自动调用频率 | 当前仍应保持手动/目标选择触发 | 不在第三阶段改成每步自动调用 |
| 审计长期分析 | 已排入 V3-J/V3-K SQLite | 不重复新增内存统计系统 |
| 风格/标签/descriptor | 已排入 V3-L/V3-M 与第四阶段 | 不在第三阶段提前拆大系统 |

第三阶段最后可补的不是新结构，而是两类轻量收口：

| 收口项 | 原因 | 是否新增阶段任务 |
|---|---|---|
| 更新过时文档状态 | 主清单曾保留早期 V3-A 提示，容易误导后续开发 | 已在本文件修正 |
| 为 V3-H/V3-I 保持测试门槛 | 新增 EmergentPresence 时必须复制 DynamicStructure 的 validator、snapshot、player 防泄漏、长跑测试路径 | 纳入 V3-H/V3-I 验收，不单独开新批次 |

结论：

```text
第三阶段不需要再临时扩展新方向。
应按 V3-H -> V3-I -> V3-J -> V3-K -> V3-L -> V3-M 推进。
AI 自由新增 descriptor 标签并落库放到第五阶段后半或更后。
```
